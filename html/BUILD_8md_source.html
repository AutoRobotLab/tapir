1 Custom non-recursive Make system
2 ================================
3 
4 Introduction
5 ------------
6 The build system uses a non-recursive approach with GNU make, similar to the
7 one documented in the paper
8 [Recursive Make Considered Harmful](http://aegis.sourceforge.net/auug97.pdf)
9 by Peter Miller, and adapting some Makefile code from
10 [here](http://evbergen.home.xs4all.nl/nonrecursive-make.html).
11 
12 The paper describes many advantages to non-recursive Make; in short, it results
13 in a cleaner and more robust dependency tree, and it means that the code can
14 quite easily be compiled in parallel.
15 
16 The system used here adds two key conveniences to the aforementioned system:
17 - Improved readability by moving standard code, especially the recursion,
18  into template files wihin the .make directory.
19  The recursion is located in [stack.mk][stack].
20 - The ability to call make from any child folder and have it behave the way
21  you would expect it to if that child folder were the root.
22 
23 General usage
24 -------------
25 Global configuration settings are set in the [root Makefile][Makefile],
26 and then individual modules can each be configured via the Makefile in their
27 own subfolder (e.g. [this][Makefile_src]).
28 
29 The subfolder Makefiles should work more or less like normal Makefiles,
30 although it is processed relative to the root folder, which means that any paths
31 should be relative to the root - the variable $(d) is provided as shorthand for
32 the relative path of the current module w.r.t. the root.
33 
34 Each module is also required to define two variables that are used for the
35 recursive processing:
36 - MODULE_NAME: The module name; should be a unique identifier. This is used
37  for generating intermediate targets for each specific module.
38 - CHILD_FOLDERS: The subdirectories that are required to be processed.
39 
40 However, in order to mainain consistency with the rest of the build system,
41 particularly with regards to build modes (release vs. debug) and with regards
42 to the structure of the build output directories (e.g. builds/release
43 and builds/debug), it is best to use the provided templates to automatically
44 generate most of the rules. Refer to
45 [the Makefile in the src folder][Makefile_src] for a simple example, or read
46 the next section for greater depth.
47 
48 Makefile templates
49 ------------------
50 The key rules that do most of the work are located in
51 [template.mk][template], which detects .cpp source files and includes
52 a default .cpp -> .o recipe, as well as automatic dependency handling.
53 
54 To provide rules on how to build the module as a whole rather than implicitly
55 via other modules that depend on it, you need to add the targets you want as
56 prerequisites for the phony build targets. For example,
57 build-solver
58 is the target that builds the entire "solver" module.
59 
60 The simplest way to define the build pattern for a module is to simply build
61 every individual object file - the rule for this is given in
62 [build.mk][build].
63 
64 The solver code is currently set to build into an archive called "libabt.a";
65 the rules to do this are defined in [the "solver" Makefile][Makefile_solver].
66 As you can see there, the rule defines all of the files that make up this
67 library in terms of the automatic OBJS\_[module] variables generated by
68 the template code.
69 
70 Problem templates
71 -----------------
72 A more complex example is the build template used for each individual
73 problem, which is located in [problem-template.mk][problem-template].
74 Its purpose is to defines recipes for the "solve" and "simulate" executables,
75 which are the key executables for running ABT on a specific problem.
76 
77 The core of this template is defining all of the dependencies for each problem.
78 The default dependencies are the other object files in the same folder as
79 solve.cpp and simulate.cpp, the solver archive libabt.a, and the external
80 library libspatialindex.
81 
82 If your problem needs to be linked to additional libraries, you can specify
83 this by adding those dependencies to the variable
84 EXTRA_LINKER_ARGS\_[module].
