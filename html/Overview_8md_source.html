11 Configuration and Building
12 --------------------------
13 This project uses a custom GNU Make build system, which is documented in further
14 detail in [this README][Build README]; the dependencies are cleanly generated
15 and make can easily be run in parallel (e.g. make -j8). The code should build OK
16 without any special configuration, but if need be the core build settings ca
17 be changed via the [root Makefile][Makefile].
18 
19 To build the core solver, simply use the command "make solver" - this will build
20 the solver as a static library in builds/release/libabt.a. Alternatively, the
21 solver can also be built as a shared library - use the command
22 "make solver CFG=shared", and the library file will be output to
23 builds/shared/libabt.so.
24 
25 The supplied code also comes with several example problems, as mentioned
26 previously. To build the code with all of the example problems, simply use
27 the command "make all" from the root directory. Alternatively, you can build
28 only a specific problem by specifying that problem as a target
29 (e.g. "make tag"), or by running the make command from within that problem's
30 source folder.
31 
32 Different build configurations are available via the "CFG" variable in the
33 Makefile, which can be set from the command line, e.g. "make CFG=shared",
34 which, as previously mentioned, builds the solver as a shared library rather
35 than via static linking. For debugging output, use "make CFG=debug", or
36 change "DEFAULT_CFG:=debug" in the [root Makefile][Makefile]. Each configuration
37 sends its output into a different directory, which will be "builds/$(CFG)".
38 
39 For additional documentation on the build system, see the
40 [build system README][Build README].

43 Usage
44 -----
45 First of all, the solver can be used for your own purposes in library form;
46 simply build it as either a static or dynamic library (as per the
47 [previous section](#configuration-and-building))
48 and then link against it.
49 
50 If you want to run the example problems, you should first compile those
51 problems, as per the [previous section](#configuration-and-building).
52 Each problem should generate two core executables - "solve" for initial offline
53 policy generation, and "simulate" for online simulation.
54 After building, these executables are placed into the directory bin/basic;
55 in order to use the default configuration settings, the executables must be run
56 from the main project folder.
57 
58 Each problem will have its own setings, which can be configured in one of two
59 ways - via command line parameters, or via configuration files. Any parameters
60 passed via command line will override the settings in the file. For a quick
61 summary of the command-line parameters available, run the executable with an
62 argument of "--help" for the usage information. More advanced settings are
63 available via config files; default versions of the configuration files are
64 located in the directory "cfg" - for example, see
65 [cfg/tag/default.cfg][Tag_config].
66 The actual settings that are available in each config file are defined by
67 subclasses of the base [Options] class, e.g. [TagOptions] specifies all of
68 the options that can be configured for Tag.

70 Implementing a new POMDP model
71 ------------------------------
72 At its core, implementation of a specific POMDP problem is done via an
73 implementation of the [Model] interface, which represents a black-box generative
74 model of a POMDP. See [TagModel] for an example implementation of this interface
75 for the Tag POMDP.
76 
77 Formally, a POMDP can be specified as an 8-tuple, (S, A, O, T, Z, R, b0, γ),
78 where
79 - S is a set of states
80 - A is a set of actions
81 - O is a set of observations
82 - T is the transition function, which is a conditional probability distribution
83  for the next state given the previous state and action,
84  i.e. T(s, a, s') = p(s' | s, a)
85 - Z is the observation function, which is a conditional probability distrubtion
86  for the observation given the action and the resulting state,
87  i.e. Z(a, o, s') = p(o | a, s')
88 - R is the reward function, which returns the expected reward for a given state
89  and action,
90  i.e. R(s, a)
91 - b0 is a probability distribution over states, which represents the agent's
92  initial knowledege.
93 - γ is the discount factor for the POMDP
94 
95 Since this is a black box model, many of these elements are defined implicitly
96 rather than explicitly. The key requirements to specify a POMDP model are:
97 - [Model::sampleAnInitState] should sample an initial state from the
98  initial belief, and hence implicitly defines b0
99 - [Model::generateStep] offers a simple generative model of the POMDP;
100  it takes a single state and action, and returns an observation, next state,
101  and a sampled reward value r, i.e. (s, a) => (o, r, s').
102  This forms an implicit definition of the transition, observation and reward
103  functions, T, Z, and R
104 - [Options::discountFactor] defines the POMDP discount factor, γ - each
105  [Model] will posses a unique instance of the [Options] class (or a subclass,
106  such as [TagOptions]).
107 - [Model::isTerminal] specifies which states will be considered terminal - the
108  method should return true for those states, and false for others.
109 
110 As for S, A, and O, the states, actions, and observations should implement
111 their respective abstract classes [State], [Action], and [Observation]. That
112 said, it is important to note that the sets of states and observations are
113 actually implicitly defined via the black-box sampling methods -
114 [Model::sampleAnInitState] should return a state within S, and
115 [Model::generateStep] should return states and observations that are within
116 S and O respectively. However, since the solver itself needs to select actions,
117 actions require more explicit treatment by the ABT algorithm. This is
118 done via the [ActionPool] interface and the method [Model::createActionPool];
119 see the [section below](#actionpool) for some additional details.
120 
121 See below for a more detailed description of the core functionality needed
122 in order to make a new problem for the ABT code.
123 
124 ### Model
125 As previously mentioned, the [Model] class is the core interface for the
126 specification of a POMDP model; see [TagModel] for a concrete example.
127 An implementation of [Model] must, at the very least, implement all of the pure
128 virtual methods in the abstract base class. These include the aforementioned
129 core methods [Model::sampleAnInitState], [Model::generateStep], and
130 [Model::isTerminal], as specified previously. A [Model] instance will also
131 possess an [Options] instance, which specifies various ABT and POMDP parameters.
132 
133 In addition to the these three core methods, and
134 [Model::createActionPool][Model::createActionPool] (see
135 [this section](#actionpool) for further details), a Model must also implement
136 [Model::sampleStateUninformed], which should generate states according to an
137 uninformed prior (as opposed to from the initial belief);
138 this is used by the default implementation for the
139 [uninformed generateParticles][Model::generateParticles2] method.
140 Custom implementations for the two particle filtering methods (
141 [informed][Model::generateParticles] and [uninformed][Model::generateParticles2]
142 ) can be written if better handling of particle depletion is needed. Also of
143 key importance (although not mandatory) is a heuristic function, which is
144 specified by [Model::getHeuristicFunction] - this uses a functional programming
145 interface, and should return a [HeuristicFunction].
146 By default, the heuristic function is simply one that always returns zero.
147 
148 If you wish the model to handle changes, see
149 [the section on changes](#implementing-changes)
150 below; in short, doing so requires implementations for several other methods,
151 most notably [Model::applyChanges].
152 
153 A convenient subclass of [Model] is [ModelWithProgramOptions], which uses
154 additional options from [SharedOptions] to provide some extra configuration
155 settings, including text-based parsing in order to select different search
156 strategies.
157 
158 ### Options
159 Each [Model] instance should possess an [Options] instance, which specifies
160 the various configuration settings to use for that problem, and for ABT to
161 use when solving that problem. The base [Options] class has several parameters
162 for ABT settings, but with respect to specifying a POMDP there are two crucial
163 values that must be specified:
164 - [Options::discountFactor] - the POMDP discount factor.
165 - [Options::numberOfStateVariables] - the number of state variables used to
166  define a state of the POMDP.
167 
168 ### State
169 Represents a state within the state space of the problem.
170 See [TagState] for an example.
171 
172 The core implementation requires only the following methods:
173 - [copy()] --- duplicates the state
174 - [hash()] --- hashes the state for storage in an std::unordered_map
175 - [equals()] --- identifies equivalent states; used in conjunction with [hash()]
176 
177 Also useful (though not required) are:
178 - [distanceTo()] --- defines a distance metric over the states;
179  the default is an infinite distance between any pair of states.
180 - [print()] --- generates a human-readable text representation
181  of the state.
182 
183 In order to be able to make changes to the policy when changes to the model
184 occur, states also need to be stored within a [StateIndex]. The default
185 implementation for this is an R\*-tree, which is implemented via a
186 [thin wrapper][RTree] around the
187 [R*-tree implementation](http://libspatialindex.github.io/overview.html#the-rtree-package)
188 from
189 [libspatialindex](http://libspatialindex.github.io).
190 In order to use this implementation, the [State] needs to implement
191 [VectorState]. In addition to the standard [State] methods, this also requires
192 the additional method [asVector()][Vector::asVector], which must return an
193 std::vector<double> representation of the state; this vector will then be stored
194 inside the [R*-tree][RTree].
195 
196 Alternatively, a custom [StateIndex] implementation can be given by
197 implementing [StateIndex] and having [Model::createStateIndex] return an
198 instance of that custom implementation.
199 
200 ### Observation
201 Represents an observation within the action space of the problem.
202 See [TagObservation] for an example.
203 Like [State], the [Observation] interface requires implementations for
204 [copy()], [equals()], and [hash()],
205 and custom implementations can be given for [distanceTo()] and [print()]
206 
207 ### Action
208 Represents an action within the action space of the problem.
209 See [TagAction] for an example.
210 Like [State], the [Action] interface requires implementations for
211 [copy()], [equals()], and [hash()],
212 and custom implementations can be given for [distanceTo()] and [print()]
213 
214 ### ActionPool
215 In order for the ABT solver to be able to search the space of possible actions,
216 it needs a way to know which actions it will need to try, and at which belief
217 nodes. This is handled in a generic manner by the [ActionPool] interface - the
218 [ActionPool] essentially works as a factory to generate individual instances of
219 [ActionMapping] for each belief node. When ABT is searching using the standard
220 [UCB search strategy], it queries the [ActionMapping] using
221 [ActionMapping::getNextActionToTry]; this essentially defines the initialization
222 phase of he UCB algorithm - as long as this method returns actions
223 (rather than nullptr), these actions will continue to be tried; moreover, it is
224 only those actions that have been tried before that will actually be selected
225 once the actual UCB algorithm is being used.
226 
227 A standard implementation of [ActionPool] that should be sufficient for most
228 purposes is provided as [EnumeratedActionPool]. This implementation assumes
229 that there is a finite, and relatively small, enumerated set of global actions
230 that can be taken from any state in the problem. Use of this implementation
231 has two prerequsites:
232 - The [Action] class used must implement the [DiscretizedPoint] interface,
233  which requires that every action be able to return its associated
234  index in the enumeration via the method [DiscretizedPoint::getBinNumber]
235 - The [EnumeratedActionPool] constructor requires, as a constructor argument,
236  a vector containing all of the actions, in the order of their enumeration.
237 
238 ### Serializer
239 If you require the ability to serialize a solver policy (e.g. to save it to
240 a file), you must provide an implementation of the abstract [Serializer]
241 class. A basic implementation that generates human-readable text representations
242 of all of the core ABT solver classes is provided by [TextSerializer], but
243 this implementation is only partial, because it doesn't come with
244 implementations for the various configurable data structures. As such,
245 in order to fully implement a [Serializer] for your problem, you will also need
246 to implement methods to serialize a few of the core interface classes - that is,
247 [State], [Action], and [Observation].
248 Additionally, the action and observation mappings also need serialization. The
249 default observation mapping is [DiscreteObservationPool], which should suffice
250 for most purposes; in order to serialize this mapping class, the serializer
251 implementation should also inherit from [DiscreteObservationTextSerializer].
252 Similarly, if you use [EnumeratedActionPool] for mapping actions, your
253 serializer should inherit from [EnumeratedActionTextSerializer].
254 
255 To see a good example of the above, have a look at [TagTextSerializer].

257 Implementing changes
258 --------------------
259 It is not mandatory for a [Model] implementation to deal with changes, but
260 if you require this functionality, the following two core implementation details
261 are the most important:
262 - A custom [ModelChange] class, e.g. [TagChange]. This class doesn't
263  have any required methods, because it's up to each individual
264  [Model] implementation to determine how it deals with changes.
265  Note that if you implement [ModelChange] you will likely also need to
266  implement serialization methods, e.g.
267  [TagTextSerializer::saveModelChange] and
268  [TagTextSerializer::loadModelChange]
269 - An implementation of [Model::applyChanges], which performs two key functions:
270  * Updating the Model's black box generators in response to the changes.
271  * Informing the Solver of the changes that have been applied. This is done
272  via the [StatePool] interface, which can be used in conjunction with
273  a custom [StateIndex] implementation, e.g. [RTree], in order to perform
274  more intelligent queries.
275  See [TagModel::applyChanges] for an example implentation.
276 
277 The [Solver] handles model changes by using a [HistoryCorrector], the default
278 implementation of which is [DefaultHistoryCorrector], which should work OK for
279 any custom problem. However, in order to use this implementation you must also
280 implement these methods of [Model][Model]:
281 - [Model::generateNextState] to generate states - an implicit definition of
282  T(s, a, s')
283 - [Model::generateObservation] to generate observations - an implicit definition
284  of Z(a, o, s')
285 - [Model::generateReward] to generate rewards - an implicit definition of
286  R(s, a)
287 
288 These three methods are used instead of [Model::generateStep] in order to avoid
289 unnecessary recalculation, and to minize the extent to which new histories
290 diverge from old ones due to re-randomization. To minimize such issues even
291 further, you can use [TransitionParameters] and [Model::generateTransition],
292 which allows you to store extra information about a generated step.
293 This can be used, for example, to store intermediate calculations,
294 or generated random numbers. See the documentation of [Model] for greater
295 detail.
