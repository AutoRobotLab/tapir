1 Toolkit for approximating and Adapting POMDP solutions In Realtime (TAPIR)
2 ==========================================================================
3 
4 TAPIR is a C++ implementation of the Adaptive Belief Tree (ABT) algorithm [1].
5 ABT is an online POMDP solver capable of adapting to modifications to the POMDP
6 model without the need to reconstruct the policy from scratch.
7 
8 [1] H. Kurniawati and V. Yadav. An Online POMDP Solver for Uncertainty Planning
9 in Dynamic Environment. Proc. Int. Symp. on Robotics Research. 2013.
10 http://robotics.itee.uq.edu.au/~hannakur/dokuwiki/papers/isrr13_abt.pdf
11 
12 For bug reports and suggestions, please email rdl.algorithm@itee.uq.edu.au
13 
14 For information on updates, please visit http://robotics.itee.uq.edu.au/~tapir

44 Configuration and Building
45 --------------------------
46 This project uses a custom GNU Make build system, which is documented in further
47 detail in [this README][.make/README.md]; the dependencies are cleanly managed
48 and make can easily be run in parallel, i.e.
49 
50  make -jN
51 
52 where N is the number of threads you want to use; 8 is good for many CPUs.
53 The code should build OK without any special configuration,
54 but if need be the core build settings can be changed via the
55 [root Makefile][Makefile].
56 
57 To build the core solver, simply use the command
58 
59  make solver
60 
61 This will build the solver as a static library in builds/release/libtapir.a.
62 Alternatively, the solver can also be built as a shared library - use the
63 command
64 
65  make solver CFG=shared
66 
67 and a shared library file will be output to builds/shared/libtapir.so
68 
69 The supplied code also comes with several example problems, as mentioned
70 previously. To build the code with all of the example problems, simply use
71 the command
72 
73  make all
74 
75 Alternatively, you can build only a specific problem by specifying that problem
76 as a target. For example,
77 
78  make tag -j4 CFG=shared
79 
80 would use 4 threads to compile; that command would build the solver as a shared
81 library, and also build the Tag problem to dynamically link against the shared
82 library file.
83 Alternatively, you can also compile a problem by running make from within that
84 problem's source folder, e.g.
85 
86  cd src/problems/tag
87  make
88 
89 Different build configurations are available via the "CFG" variable in the
90 Makefile, which can be set from the command line; for example, debugging
91 output can be generated by using
92 
93  make CFG=debug
94 
95 or by setting
96 
97  DEFAULT_CFG:=debug
98 
99 in the [root Makefile][Makefile]. Each configuration sends its output into a
100 different directory, which will be "builds/$(CFG)".
101 
102 For additional documentation on the build system, see the
103 [build system README][.make/README.md].

106 Usage
107 -----
108 First of all, the solver can be used for your own purposes in library form;
109 simply build it as either a static or dynamic library (as per the
110 [previous section](#configuration-and-building))
111 and then link against it.
112 
113 If you want to run the example problems, you should first compile those
114 problems, as per the [previous section](#configuration-and-building).
115 Each problem should generate two core executables - "solve" for initial offline
116 policy generation, and "simulate" for online simulation.
117 After building, these executables are placed into the directory bin/basic;
118 in order to use the default configuration settings, the executables must be run
119 from the main project folder.
120 
121 Each problem will have its own setings, which can be configured in one of two
122 ways - via command line parameters, or via configuration files. Any parameters
123 passed via command line will override the settings in the file. For a quick
124 summary of the command-line parameters available, run the executable with an
125 argument of "--help" for the usage information. More advanced settings are
126 available via config files; default versions of the configuration files are
127 located in the directory "cfg" - for example, see
128 [cfg/tag/default.cfg][Tag_config].
129 The actual settings that are available in each config file are defined by
130 subclasses of the base [Options] class, e.g. [TagOptions] specifies all of
131 the options that can be configured for Tag.

133 Implementing a new POMDP model
134 ------------------------------
135 At its core, implementation of a specific POMDP problem is done via an
136 implementation of the [Model] interface, which represents a black-box generative
137 model of a POMDP. See [TagModel] for an example implementation of this interface
138 for the Tag POMDP.
139 
140 Formally, a POMDP can be specified as an 8-tuple, (S, A, O, T, Z, R, b0, γ),
141 where
142 - S is a set of states
143 - A is a set of actions
144 - O is a set of observations
145 - T is the transition function, which is a conditional probability distribution
146  for the next state given the previous state and action,
147  i.e. T(s, a, s') = p(s' | s, a)
148 - Z is the observation function, which is a conditional probability distrubtion
149  for the observation given the action and the resulting state,
150  i.e. Z(a, o, s') = p(o | a, s')
151 - R is the reward function, which returns the expected reward for a given state
152  and action,
153  i.e. R(s, a)
154 - b0 is a probability distribution over states, which represents the agent's
155  initial knowledege.
156 - γ is the discount factor for the POMDP
157 
158 Since this is a black box model, many of these elements are defined implicitly
159 rather than explicitly. The key requirements to specify a POMDP model are:
160 - [Model::sampleAnInitState] should sample an initial state from the
161  initial belief, and hence implicitly defines b0
162 - [Model::generateStep] offers a simple generative model of the POMDP;
163  it takes a single state and action, and returns an observation, next state,
164  and a sampled reward value r, i.e. (s, a) => (o, r, s').
165  This forms an implicit definition of the transition, observation and reward
166  functions, T, Z, and R
167 - [Options::discountFactor] defines the POMDP discount factor, γ - each
168  [Model] will posses a unique instance of the [Options] class (or a subclass,
169  such as [TagOptions]).
170 - [Model::isTerminal] specifies which states will be considered terminal - the
171  method should return true for those states, and false for others.
172 
173 As for S, A, and O, the states, actions, and observations should implement
174 their respective abstract classes [State], [Action], and [Observation]. That
175 said, it is important to note that the sets of states and observations are
176 actually implicitly defined via the black-box sampling methods -
177 [Model::sampleAnInitState] should return a state within S, and
178 [Model::generateStep] should return states and observations that are within
179 S and O respectively. However, since the solver itself needs to select actions,
180 actions require more explicit treatment by the ABT algorithm. This is
181 done via the [ActionPool] interface and the method [Model::createActionPool];
182 see the [section below](#actionpool) for some additional details.
183 
184 See below for a more detailed description of the core functionality needed
185 in order to make a new problem for TAPIR.
186 
187 ### Model
188 As previously mentioned, the [Model] class is the core interface for the
189 specification of a POMDP model; see [TagModel] for a concrete example.
190 An implementation of [Model] must, at the very least, implement all of the pure
191 virtual methods in the abstract base class. These include the aforementioned
192 core methods [Model::sampleAnInitState], [Model::generateStep], and
193 [Model::isTerminal], as specified previously. A [Model] instance will also
194 possess an [Options] instance, which specifies various ABT and POMDP parameters.
195 
196 In addition to the these three core methods, and
197 [Model::createActionPool][Model::createActionPool] (see
198 [this section](#actionpool) for further details), a Model must also implement
199 [Model::sampleStateUninformed], which should generate states according to an
200 uninformed prior (as opposed to from the initial belief);
201 this is used by the default implementation for the
202 [uninformed generateParticles][Model::generateParticles2] method.
203 Custom implementations for the two particle filtering methods (
204 [informed][Model::generateParticles] and [uninformed][Model::generateParticles2]
205 ) can be written if better handling of particle depletion is needed. Also of
206 key importance (although not mandatory) is a heuristic function, which is
207 specified by [Model::getHeuristicFunction] - this uses a functional programming
208 interface, and should return a [HeuristicFunction].
209 By default, the heuristic function is simply one that always returns zero.
210 
211 If you wish the model to handle changes, see
212 [the section on changes](#implementing-changes)
213 below; in short, doing so requires implementations for several other methods,
214 most notably [Model::applyChanges].
215 
216 A convenient subclass of [Model] is [ModelWithProgramOptions], which uses
217 additional options from [SharedOptions] to provide some extra configuration
218 settings, including text-based parsing in order to select different search
219 strategies.
220 
221 ### Options
222 Each [Model] instance should possess an [Options] instance, which specifies
223 the various configuration settings to use for that problem, and for TAPIR to
224 use when solving that problem. The base [Options] class has several parameters
225 for ABT settings, but with respect to specifying a POMDP there are two crucial
226 values that must be specified:
227 - [Options::discountFactor] - the POMDP discount factor.
228 - [Options::numberOfStateVariables] - the number of state variables used to
229  define a state of the POMDP.
230 
231 ### State
232 Represents a state within the state space of the problem.
233 See [TagState] for an example.
234 
235 The core implementation requires only the following methods:
236 - [copy()] --- duplicates the state
237 - [hash()] --- hashes the state for storage in an std::unordered_map
238 - [equals()] --- identifies equivalent states; used in conjunction with [hash()]
239 
240 Also useful (though not required) are:
241 - [distanceTo()] --- defines a distance metric over the states;
242  the default is an infinite distance between any pair of states.
243 - [print()] --- generates a human-readable text representation
244  of the state.
245 
246 In order to be able to make changes to the policy when changes to the model
247 occur, states also need to be stored within a [StateIndex]. The default
248 implementation for this is an R\*-tree, which is implemented via a
249 [thin wrapper][RTree] around the
250 [R*-tree implementation](http://libspatialindex.github.io/overview.html#the-rtree-package)
251 from
252 [libspatialindex](http://libspatialindex.github.io).
253 In order to use this implementation, the [State] needs to implement
254 [VectorState]. In addition to the standard [State] methods, this also requires
255 the additional method [asVector()][Vector::asVector], which must return an
256 std::vector<double> representation of the state; this vector will then be stored
257 inside the [R*-tree][RTree].
258 
259 Alternatively, a custom [StateIndex] implementation can be given by
260 implementing [StateIndex] and having [Model::createStateIndex] return an
261 instance of that custom implementation.
262 
263 ### Observation
264 Represents an observation within the action space of the problem.
265 See [TagObservation] for an example.
266 Like [State], the [Observation] interface requires implementations for
267 [copy()], [equals()], and [hash()],
268 and custom implementations can be given for [distanceTo()] and [print()]
269 
270 ### Action
271 Represents an action within the action space of the problem.
272 See [TagAction] for an example.
273 Like [State], the [Action] interface requires implementations for
274 [copy()], [equals()], and [hash()],
275 and custom implementations can be given for [distanceTo()] and [print()]

277 ### ActionPool
278 In order for the ABT solver to be able to search the space of possible actions,
279 it needs a way to know which actions it will need to try, and at which belief
280 nodes. This is handled in a generic manner by the [ActionPool] interface - the
281 [ActionPool] essentially works as a factory to generate individual instances of
282 [ActionMapping] for each belief node. When ABT is searching using the standard
283 [UCB search strategy], it queries the [ActionMapping] using
284 [ActionMapping::getNextActionToTry]; this essentially defines the initialization
285 phase of he UCB algorithm - as long as this method returns actions
286 (rather than nullptr), these actions will continue to be tried; moreover, it is
287 only those actions that have been tried before that will actually be selected
288 once the actual UCB algorithm is being used.
289 
290 A standard implementation of [ActionPool] that should be sufficient for most
291 purposes is provided as [EnumeratedActionPool]. This implementation assumes
292 that there is a finite, and relatively small, enumerated set of global actions
293 that can be taken from any state in the problem. Use of this implementation
294 has two prerequsites:
295 - The [Action] class used must implement the [DiscretizedPoint] interface,
296  which requires that every action be able to return its associated
297  index in the enumeration via the method [DiscretizedPoint::getBinNumber]
298 - The [EnumeratedActionPool] constructor requires, as a constructor argument,
299  a vector containing all of the actions, in the order of their enumeration.
300 
301 ### Serializer
302 If you require the ability to serialize a solver policy (e.g. to save it to
303 a file), you must provide an implementation of the abstract [Serializer]
304 class. A basic implementation that generates human-readable text representations
305 of all of the core ABT solver classes is provided by [TextSerializer], but
306 this implementation is only partial, because it doesn't come with
307 implementations for the various configurable data structures. As such,
308 in order to fully implement a [Serializer] for your problem, you will also need
309 to implement methods to serialize a few of the core interface classes - that is,
310 [State], [Action], and [Observation].
311 Additionally, the action and observation mappings also need serialization. The
312 default observation mapping is [DiscreteObservationPool], which should suffice
313 for most purposes; in order to serialize this mapping class, the serializer
314 implementation should also inherit from [DiscreteObservationTextSerializer].
315 Similarly, if you use [EnumeratedActionPool] for mapping actions, your
316 serializer should inherit from [EnumeratedActionTextSerializer].
317 
318 To see a good example of the above, have a look at [TagTextSerializer].

320 Implementing changes
321 --------------------
322 It is not mandatory for a [Model] implementation to deal with changes, but
323 if you require this functionality, the following two core implementation details
324 are the most important:
325 - A custom [ModelChange] class, e.g. [TagChange]. This class doesn't
326  have any required methods, because it's up to each individual
327  [Model] implementation to determine how it deals with changes.
328  Note that if you implement [ModelChange] you will likely also need to
329  implement serialization methods, e.g.
330  [TagTextSerializer::saveModelChange] and
331  [TagTextSerializer::loadModelChange]
332 - An implementation of [Model::applyChanges], which performs two key functions:
333  * Updating the Model's black box generators in response to the changes.
334  * Informing the Solver of the changes that have been applied. This is done
335  via the [StatePool] interface, which can be used in conjunction with
336  a custom [StateIndex] implementation, e.g. [RTree], in order to perform
337  more intelligent queries.
338  See [TagModel::applyChanges] for an example implentation.
339 
340 The [Solver] handles model changes by using a [HistoryCorrector], the default
341 implementation of which is [DefaultHistoryCorrector], which should work OK for
342 any custom problem. However, in order to use this implementation you must also
343 implement these methods of [Model][Model]:
344 - [Model::generateNextState] to generate states - an implicit definition of
345  T(s, a, s')
346 - [Model::generateObservation] to generate observations - an implicit definition
347  of Z(a, o, s')
348 - [Model::generateReward] to generate rewards - an implicit definition of
349  R(s, a)
350 
351 These three methods are used instead of [Model::generateStep] in order to avoid
352 unnecessary recalculation, and to minize the extent to which new histories
353 diverge from old ones due to re-randomization. To minimize such issues even
354 further, you can use [TransitionParameters] and [Model::generateTransition],
355 which allows you to store extra information about a generated step.
356 This can be used, for example, to store intermediate calculations,
357 or generated random numbers. See the documentation of [Model] for greater
358 detail.
