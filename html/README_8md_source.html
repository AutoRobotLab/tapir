3 abt - Adaptive Belief Tree {#abt}
4 ==========================
5 
6 Introduction {#introduction}
7 ------------
8 An online POMDP solver capable of adapting to modifications to the POMDP model
9 without the need to reconstruct the policy from scratch. The solver uses
10 informed sampling methods to generate a sampled belief tree, which embodies the
11 current policy, while also keeping track of all of the sampled trajectories
12 in order to allow for efficient updates in response to changes to the
13 POMDP model.
14 
15 Code structure and examples {#code-structure-and-examples}
16 ---------------------------
17 The core solver code is located in src/solver, while several example problems
18 are given in src/problems:
19 - [Tag]
20 - [RockSample]

22 System Requirements {#system-requirements}
23 -------------------
24 The code is written primarily for Linux environments, although it should also
25 build OK in Windows.

27 Building and running the C++ source code requires the following, which should
28 all be available via a package manager; Debian/Ubuntu package names are given

34 Configuration and Building {#configuration-and-building}
35 --------------------------
36 This project uses a custom GNU Make build system, which is documented in further
37 detail in [this README][Build README]; the dependencies are cleanly generated
38 and make can easily be run in parallel (e.g. make -j8). The code should build OK
39 without any special configuration, but if need be the core build settings ca
40 be changed via the [root Makefile][Makefile].
41 
42 To build the core solver, simply use the command "make solver" - this will build
43 the solver as a static library in builds/release/libabt.a. Alternatively, the
44 solver can also be built as a shared library - use the command
45 "make solver CFG=shared", and the library file will be output to
46 builds/shared/libabt.so.
47 
48 The supplied code also comes with several example problems, as mentioned
49 previously. To build the code with all of the example problems, simply use
50 the command "make all" from the root directory. Alternatively, you can build
51 only a specific problem by specifying that problem as a target
52 (e.g. "make tag"), or by running the make command from within that problem's
53 source folder.
54 
55 Different build configurations are available via the "CFG" variable in the
56 Makefile, which can be set from the command line, e.g. "make CFG=shared",
57 which, as previously mentioned, builds the solver as a shared library rather
58 than via static linking. For debugging output, use "make CFG=debug", or
59 change "DEFAULT_CFG:=debug" in the [root Makefile][Makefile]. Each configuration
60 sends its output into a different directory, which will be "builds/$(CFG)".
61 
62 For additional documentation on the build system, see the
63 [build system README][Build README].

65 Usage {#usage}
66 -----
67 First of all, the solver can be used for your own purposes in library form;
68 simply build it as either a static or dynamic library (as per the
69 [previous section](#configuration-and-building))
70 and then link against it.
71 
72 If you want to run the example problems, you should first compile those
73 problems, as per the [previous section](#configuration-and-building).
74 Each problem should generate two core executables - "solve" for initial offline
75 policy generation, and "simulate" for online simulation.
76 After building, these executables are placed into the directory bin/basic;
77 in order to use the default configuration settings, the executables must be run
78 from the main project folder.
79 
80 Each problem will have its own setings, which can be configured in one of two
81 ways - via command line parameters, or via configuration files. Any parameters
82 passed via command line will override the settings in the file. For a quick
83 summary of the command-line parameters available, run the executable with an
84 argument of "--help" for the usage information. More advanced settings are
85 available via config files; default versions of the configuration files are
86 located in the directory "cfg" - for example, see
87 [cfg/tag/default.cfg][Tag_config].
88 The actual settings that are available in each config file are defined by
89 subclasses of the base [Options] class, e.g. [TagOptions] specifies all of
90 the options that can be configured for Tag.

92 Implementing a new POMDP model {#implementing-a-new-pomdp-model}
93 ------------------------------
94 At its core, implementation of a specific POMDP problem is done via an
95 implementation of the [Model] interface, which represents a black-box generative
96 model of a POMDP. See [TagModel] for an example implementation of this interface
97 for the Tag POMDP.
98 
99 Formally, a POMDP can be specified as an 8-tuple, (S, A, O, T, Z, R, b0, γ),
100 where
101 - S is a set of states
102 - A is a set of actions
103 - O is a set of observations
104 - T is the transition function, which is a conditional probability distribution
105  for the next state given the previous state and action,
106  i.e. T(s, a, s') = p(s' | s, a)
107 - Z is the observation function, which is a conditional probability distrubtion
108  for the observation given the action and the resulting state,
109  i.e. Z(a, o, s') = p(o | a, s')
110 - R is the reward function, which returns the expected reward for a given state
111  and action,
112  i.e. R(s, a)
113 - b0 is a probability distribution over states, which represents the agent's
114  initial knowledege.
115 - γ is the discount factor for the POMDP
116 
117 Since this is a black box model, many of these elements are defined implicitly
118 rather than explicitly. The key requirements to specify a POMDP model are:
119 - [Model::sampleAnInitState] should sample an initial state from the
120  initial belief, and hence implicitly defines b0
121 - [Model::generateStep] offers a simple generative model of the POMDP;
122  it takes a single state and action, and returns an observation, next state,
123  and a sampled reward value r, i.e. (s, a) => (o, r, s').
124  This forms an implicit definition of the transition, observation and reward
125  functions, T, Z, and R
126 - [Options::discountFactor] defines the POMDP discount factor, γ - each
127  [Model] will posses a unique instance of the [Options] class (or a subclass,
128  such as [TagOptions]).
129 - [Model::isTerminal] specifies which states will be considered terminal - the
130  method should return true for those states, and false for others.
131 
132 As for S, A, and O, the states, actions, and observations should implement
133 their respective abstract classes [State], [Action], and [Observation]. That
134 said, it is important to note that the sets of states and observations are
135 actually implicitly defined via the black-box sampling methods -
136 [Model::sampleAnInitState] should return a state within S, and
137 [Model::generateStep] should return states and observations that are within
138 S and O respectively. However, since the solver itself needs to select actions,
139 actions require more explicit treatment by the ABT algorithm. This is
140 done via the [ActionPool] interface and the method [Model::createActionPool];
141 see the [section below](#actionpool) for some additional details.
142 
143 See below for a more detailed description of the core functionality needed
144 in order to make a new problem for the ABT code.
145 
146 ### Model {#model}
147 As previously mentioned, the [Model] class is the core interface for the
148 specification of a POMDP model; see [TagModel] for a concrete example.
149 An implementation of [Model] must, at the very least, implement all of the pure
150 virtual methods in the abstract base class. These include the aforementioned
151 core methods [Model::sampleAnInitState], [Model::generateStep], and
152 [Model::isTerminal], as specified previously. A [Model] instance will also
153 possess an [Options] instance, which specifies various ABT and POMDP parameters.
154 
155 In addition to the these three core methods, and
156 [Model::createActionPool][Model::createActionPool] (see
157 [this section](#actionpool) for further details), a Model must also implement
158 [Model::sampleStateUninformed], which should generate states according to an
159 uninformed prior (as opposed to from the initial belief);
160 this is used by the default implementation for the
161 [uninformed generateParticles][Model::generateParticles2] method.
162 Custom implementations for the two particle filtering methods (
163 [informed][Model::generateParticles] and [uninformed][Model::generateParticles2]
164 ) can be written if better handling of particle depletion is needed. Also of
165 key importance (although not mandatory) is a heuristic function, which is
166 specified by [Model::getHeuristicFunction] - this uses a functional programming
167 interface, and should return a [HeuristicFunction].
168 By default, the heuristic function is simply one that always returns zero.
169 
170 If you wish the model to handle changes, see
171 [the section on changes](#implementing-changes)
172 below; in short, doing so requires implementations for several other methods,
173 most notably [Model::applyChanges].
174 
175 A convenient subclass of [Model] is [ModelWithProgramOptions], which uses
176 additional options from [SharedOptions] to provide some extra configuration
177 settings, including text-based parsing in order to select different search
178 strategies.
179 
180 ### Options {#options}
181 Each [Model] instance should possess an [Options] instance, which specifies
182 the various configuration settings to use for that problem, and for ABT to
183 use when solving that problem. The base [Options] class has several parameters
184 for ABT settings, but with respect to specifying a POMDP there are two crucial
185 values that must be specified:
186 - [Options::discountFactor] - the POMDP discount factor.
187 - [Options::numberOfStateVariables] - the number of state variables used to
188  define a state of the POMDP.
189 
190 ### State {#state}
191 Represents a state within the state space of the problem.
192 See [TagState] for an example.
193 
194 The core implementation requires only the following methods:
195 - [copy()] --- duplicates the state
196 - [hash()] --- hashes the state for storage in an std::unordered_map
197 - [equals()] --- identifies equivalent states; used in conjunction with [hash()]
198 
199 Also useful (though not required) are:
200 - [distanceTo()] --- defines a distance metric over the states;
201  the default is an infinite distance between any pair of states.
202 - [print()] --- generates a human-readable text representation
203  of the state.
204 
205 In order to be able to make changes to the policy when changes to the model
206 occur, states also need to be stored within a [StateIndex]. The default
207 implementation for this is an R\*-tree, which is implemented via a
208 [thin wrapper][RTree] around the
209 [R*-tree implementation](http://libspatialindex.github.io/overview.html#the-rtree-package)
210 from
211 [libspatialindex](http://libspatialindex.github.io).
212 In order to use this implementation, the [State] needs to implement
213 [VectorState]. In addition to the standard [State] methods, this also requires
214 the additional method [asVector()][Vector::asVector], which must return an
215 std::vector<double> representation of the state; this vector will then be stored
216 inside the [R*-tree][RTree].
217 
218 Alternatively, a custom [StateIndex] implementation can be given by
219 implementing [StateIndex] and having [Model::createStateIndex] return an
220 instance of that custom implementation.
221 
222 ### Observation {#observation}
223 Represents an observation within the action space of the problem.
224 See [TagObservation] for an example.
225 Like [State], the [Observation] interface requires implementations for
226 [copy()], [equals()], and [hash()],
227 and custom implementations can be given for [distanceTo()] and [print()]
228 
229 ### Action {#action}
230 Represents an action within the action space of the problem.
231 See [TagAction] for an example.
232 Like [State], the [Action] interface requires implementations for
233 [copy()], [equals()], and [hash()],
234 and custom implementations can be given for [distanceTo()] and [print()]
235 
236 ### ActionPool {#actionpool}
237 In order for the ABT solver to be able to search the space of possible actions,
238 it needs a way to know which actions it will need to try, and at which belief
239 nodes. This is handled in a generic manner by the [ActionPool] interface - the
240 [ActionPool] essentially works as a factory to generate individual instances of
241 [ActionMapping] for each belief node. When ABT is searching using the standard
242 [UCB search strategy], it queries the [ActionMapping] using
243 [ActionMapping::getNextActionToTry]; this essentially defines the initialization
244 phase of he UCB algorithm - as long as this method returns actions
245 (rather than nullptr), these actions will continue to be tried; moreover, it is
246 only those actions that have been tried before that will actually be selected
247 once the actual UCB algorithm is being used.
248 
249 A standard implementation of [ActionPool] that should be sufficient for most
250 purposes is provided as [EnumeratedActionPool]. This implementation assumes
251 that there is a finite, and relatively small, enumerated set of global actions
252 that can be taken from any state in the problem. Use of this implementation
253 has two prerequsites:
254 - The [Action] class used must implement the [DiscretizedPoint] interface,
255  which requires that every action be able to return its associated
256  index in the enumeration via the method [DiscretizedPoint::getBinNumber]
257 - The [EnumeratedActionPool] constructor requires, as a constructor argument,
258  a vector containing all of the actions, in the order of their enumeration.
259 
260 ### Serializer {#serializer}
261 If you require the ability to serialize a solver policy (e.g. to save it to
262 a file), you must provide an implementation of the abstract [Serializer]
263 class. A basic implementation that generates human-readable text representations
264 of all of the core ABT solver classes is provided by [TextSerializer], but
265 this implementation is only partial, because it doesn't come with
266 implementations for the various configurable data structures. As such,
267 in order to fully implement a [Serializer] for your problem, you will also need
268 to implement methods to serialize a few of the core interface classes - that is,
269 [State], [Action], and [Observation].
270 Additionally, the action and observation mappings also need serialization. The
271 default observation mapping is [DiscreteObservationPool], which should suffice
272 for most purposes; in order to serialize this mapping class, the serializer
273 implementation should also inherit from [DiscreteObservationTextSerializer].
274 Similarly, if you use [EnumeratedActionPool] for mapping actions, your
275 serializer should inherit from [EnumeratedActionTextSerializer].
276 
277 To see a good example of the above, have a look at [TagTextSerializer].

279 Implementing changes {#implementing-changes}
280 --------------------
281 It is not mandatory for a [Model] implementation to deal with changes, but
282 if you require this functionality, the following two core implementation details
283 are the most important:
284 - A custom [ModelChange] class, e.g. [TagChange]. This class doesn't
285  have any required methods, because it's up to each individual
286  [Model] implementation to determine how it deals with changes.
287  Note that if you implement [ModelChange] you will likely also need to
288  implement serialization methods, e.g.
289  [TagTextSerializer::saveModelChange] and
290  [TagTextSerializer::loadModelChange]
291 - An implementation of [Model::applyChanges], which performs two key functions:
292  * Updating the Model's black box generators in response to the changes.
293  * Informing the Solver of the changes that have been applied. This is done
294  via the [StatePool] interface, which can be used in conjunction with
295  a custom [StateIndex] implementation, e.g. [RTree], in order to perform
296  more intelligent queries.
297  See [TagModel::applyChanges] for an example implentation.
298 
299 The [Solver] handles model changes by using a [HistoryCorrector], the default
300 implementation of which is [DefaultHistoryCorrector], which should work OK for
301 any custom problem. However, in order to use this implementation you must also
302 implement these methods of [Model][Model]:
303 - [Model::generateNextState] to generate states - an implicit definition of
304  T(s, a, s')
305 - [Model::generateObservation] to generate observations - an implicit definition
306  of Z(a, o, s')
307 - [Model::generateReward] to generate rewards - an implicit definition of
308  R(s, a)
309 
310 These three methods are used instead of [Model::generateStep] in order to avoid
311 unnecessary recalculation, and to minize the extent to which new histories
312 diverge from old ones due to re-randomization. To minimize such issues even
313 further, you can use [TransitionParameters] and [Model::generateTransition],
314 which allows you to store extra information about a generated step.
315 This can be used, for example, to store intermediate calculations,
316 or generated random numbers. See the documentation of [Model] for greater
317 detail.
